## PHASE 7: Docker Support

---

Add Docker support to DoochyBot at /home/san/DoochyBot/. The bot should run in a container with persistent data and easy deployment. Use pnpm, not npm.

FILES TO CREATE

1. Dockerfile

Multi-stage build for small final image.

```dockerfile
FROM node:20-alpine AS base
RUN corepack enable && corepack prepare pnpm@latest --activate
WORKDIR /app
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile --prod

FROM node:20-alpine
RUN apk add --no-cache dumb-init
RUN corepack enable && corepack prepare pnpm@latest --activate
WORKDIR /app
COPY --from=base /app/node_modules ./node_modules
COPY . .
RUN mkdir -p /app/data
EXPOSE 9009
ENTRYPOINT ["dumb-init", "--"]
CMD ["pnpm", "start"]
```

Why:
- dumb-init handles PID 1 properly (signal forwarding, zombie reaping)
- Two-stage build keeps the image small (no dev dependencies in final stage)
- pnpm for consistent, fast installs
- data/ directory created at build time
- No port 9009 needed anymore (no Express proxy) but keep it for future use

2. .dockerignore

Prevent local files from leaking into the image:

```
node_modules
data
.env
.env.docker
*.log
.git
.gitignore
README.md
SETUP.md
setup.js
```

Why:
- node_modules from host must not be copied (wrong architecture, dev deps)
- data/ contains runtime state, mounted as volume
- .env has secrets, passed via docker-compose or --env-file
- setup.js is for host use only, not needed in container

3. docker-compose.yml

Easy one-command deployment:

```yaml
version: '3.8'

services:
  doochybot:
    build: .
    image: doochybot:latest
    container_name: doochybot
    restart: unless-stopped
    env_file:
      - .env.docker
    volumes:
      - ./data:/app/data
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
```

Why:
- restart: unless-stopped — bot comes back after crashes or host reboot
- .env.docker used instead of .env — keeps Docker config separate
- data/ mounted as volume — trade history and settings survive container rebuilds
- Log rotation prevents disk fill from bot logs

4. Package.json scripts

Add Docker-specific scripts:

```json
{
  "scripts": {
    "start": "node src/index.js",
    "setup": "node setup.js",
    "docker:build": "docker build -t doochybot .",
    "docker:up": "docker-compose up -d",
    "docker:down": "docker-compose down",
    "docker:logs": "docker-compose logs -f",
    "docker:restart": "docker-compose restart"
  }
}
```

5. .env.docker

This file is generated by setup.js (Phase 6) alongside .env. It contains identical values. For Docker, the env_file directive in docker-compose.yml reads this file.

Add a comment at the top of .env.docker explaining its purpose:

```
# DoochyBot Docker Configuration
# Generated by setup.js — do not edit manually
# Used by docker-compose.yml
```

6. Ensure the bot works in Docker

Check src/index.js for any hardcoded paths that might break in a container:

- All file paths must use path.join() relative to process.cwd() or __dirname
- The data/ directory is at /app/data inside the container
- src/storage.js must reference data/ relative to the project root, not an absolute path
- If any path is hardcoded like /home/san/..., it will fail in Docker

Fix: In storage.js and any other file that accesses the filesystem, use:

```javascript
const path = require('path');
const DATA_DIR = path.join(__dirname, '..', '..', 'data');
// or
const DATA_DIR = path.join(process.cwd(), 'data');
```

Check all files for path issues:
- src/storage.js — settings.json and tradeLog.jsonl paths
- Any other file reading/writing to disk

TESTING

1. Run setup.js to generate .env and .env.docker with real credentials
2. Build: pnpm run docker:build
3. Start: pnpm run docker:up
4. Check logs: pnpm run docker:logs — should see "DoochyBot ready"
5. Verify data persistence: restart container, check that settings and trade history survived
6. Verify Telegram commands work
7. Stop: pnpm run docker:down

QUICK START FOR NEW USERS

After this phase, a new user's flow is:

```bash
git clone <repo>
cd DoochyBot
pnpm install
pnpm run setup     # interactive wizard, writes .env + .env.docker
pnpm run docker:build
pnpm run docker:up
```

That's four commands to go from zero to running bot.

IMPORTANT

- The Docker image must use the exact same Node version as development (Node 20)
- pnpm must be available in both build stages
- The .dockerignore must exclude .env (host secrets) but .env.docker is passed explicitly
- Log to stdout in Docker (console.log already does this) — Docker captures stdout
- No process managers inside the container (no PM2, no supervisor) — Docker handles restarts
- The container runs as root (acceptable for a personal trading bot, not a multi-tenant service)
- dumb-init is the only process manager — it's 1MB and handles signals correctly
- Health checks: Docker can poll /health if we add an HTTP health endpoint. Not required for now — the bot logs its status on startup.
